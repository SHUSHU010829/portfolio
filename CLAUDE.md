# About this Project

這是一個介紹我自己順便放上履歷的專案。

## 語言規範

- 回應語言：Always respond in 中文
- 註釋/文案：沒有特殊說明時，默認使用英文

## 程式碼設計原則

- 簡潔性：不要過度設計，保證程式碼簡潔易懂，簡單實用
- 可讀性：程式碼要清晰而非聰明，適度添加註解
- 模塊化：注意模塊設計，盡量使用設計模式
- 複雜度控制：注意圈複雜度，程式碼盡可能複用
- 提取公共函數：將重複邏輯提取為函數
- 避免硬編碼：要實現真正的邏輯
- 最小化修改：改動時盡量不要修改到其他模塊的程式碼

## 程式碼品質

- 健壯性：程式碼要健壯、可維護、可拓展
- 錯誤處理：增加 try, catch 錯誤處理
- 測試：請務必測試

## 組件架構規範

- 組件分層架構：業務層（Business Layer）→ 邏輯層（Logic Layer）→ 公用底層封裝（Common Base Layer）
- 組件複用原則：
  - **最多複用表單層級組件**（Form-level Components）
  - **千萬不要複用頁面級組件**（Page-level Components）
  - 頁面級組件保持獨立，避免跨頁面複用造成的耦合問題

## TypeScript 規範

- `any` 使用規範（分層策略）：
  - **公用底層封裝**：適度使用 `any` 是可以接受的（如泛型約束、動態類型處理等場景）
  - **邏輯層**：慎用 `any`，優先使用明確類型或泛型
  - **業務層**：嚴禁使用 `any`，必須使用明確的類型定義

## React/Next.js 規範

- 預設 Server Components：默認使用 React Server Components，僅在需要交互時使用 'use client'
- 圖片加載：盡可能使用 Next/Image
- Server Actions：必須返回結構化的 {success: boolean, message: string}
- TypeScript 嚴格模式

## UI/UX 設計規範

- 響應式設計：介面設計要兼顧多裝置響應式
- Tailwind 簡潔：保持 Tailwind className 簡潔

## 開發伺服器規範

- **禁止自動執行 `npm run dev`**：除非用戶明確要求，否則不要主動運行開發伺服器
- 測試驗證：如需測試功能，應先詢問用戶是否需要啟動開發伺服器
- 背景進程管理：如果開發伺服器已在背景運行，應適時提醒用戶並詢問是否需要停止

## 工作流程

- 任務執行前：先條列出所有要完成的步驟清單，再開始進行，執行中非必要問題就不需要詢問了，只需執行完畢後報告剛剛調整了什麼即可
- 需求審查：如果需求不合理，請直接告知
